# 技術選定

## 1. 前提整理

`specification.md` から読み取れる主要要件は以下です。

- ローカルで動くアプリケーションであること
- 入力は `mp4`
- 動画読込時にフレーム変換を先に実行すること
- 変換後の各フレームに対して直接描画できること
- 塗りつぶしは単一フレームではなく、閉領域が続くフレーム区間へ伝播すること
- 最終的に `mp4` として書き出せること

この要件だと、単なる動画プレイヤーではなく、`動画前処理` と `フレーム編集` を持つローカル編集アプリとして設計するのが妥当です。

## 2. 選定方針

評価軸は以下に置きます。

1. ローカル配布しやすいこと
2. 動画の入出力が安定していること
3. 線画変換とフレーム横断の塗りつぶしを実装しやすいこと
4. UI 実装速度が高いこと
5. 将来的に性能改善しやすいこと

## 3. デスクトップアプリ基盤

### 候補

#### A. Electron + TypeScript

- 利点
  - Web 技術で UI を作りやすい
  - 実績が多い
- 欠点
  - Chromium 同梱前提でアプリサイズが大きくなりやすい
  - 動画前処理のような重い処理を安全に分離すると、メイン/レンダラ/ユーティリティプロセスの整理がやや重い

#### B. Tauri + TypeScript + Rust

- 利点
  - Web UI を使いながら、重い処理を Rust 側に逃がせる
  - 外部バイナリを sidecar として同梱でき、`ffmpeg` を配布に含めやすい
  - ローカルアプリとして軽量にまとめやすい
- 欠点
  - Electron より情報量は少ない
  - Rust 側実装の学習コストがある

#### C. ネイティブ GUI + C++/Qt

- 利点
  - 性能面では強い
  - OpenCV や FFmpeg と近い距離で組める
- 欠点
  - UI 実装コストが高い
  - 初期開発速度が落ちやすい

### 採用

`Tauri + TypeScript + Rust`

### 選定理由

- このアプリは UI は 2D エディタ寄りだが、コア難所は動画処理であり、そこだけを Rust に寄せられる構成が最も合理的
- `ffmpeg` を sidecar として同梱できるため、ユーザー環境への追加インストールを避けやすい
- Electron より配布サイズと常駐負荷を抑えやすい
- 仕様上「ローカルで動く」が重要であり、Web サービス化前提の構成にする必要がない

## 4. 動画入出力と前処理

### 候補

#### A. FFmpeg のみで完結

- 利点
  - `mp4` のデコード/エンコードは非常に強い
  - 画像シーケンス化も安定している
- 欠点
  - 線画変換の画作りやフレーム横断処理の自由度が低い
  - 「昔のお絵かき BBS っぽい線画」を細かく追い込むにはつらい

#### B. OpenCV のみで完結

- 利点
  - 画像処理アルゴリズムは組みやすい
- 欠点
  - `mp4` の入出力や配布時のコーデック面は FFmpeg より読みにくい
  - 出力品質と扱える形式の面で別途考慮が増える

#### C. FFmpeg + OpenCV

- 利点
  - FFmpeg に動画入出力を任せ、OpenCV に線画抽出や領域解析を任せられる
  - 役割分担が明確
  - 将来、前処理アルゴリズムを差し替えやすい
- 欠点
  - 実装コンポーネントは増える

### 採用

`FFmpeg + OpenCV`

### 選定理由

- `mp4` の読込・書出しは FFmpeg に任せるのが最も堅い
- 線画変換は OpenCV の `Canny`、平滑化、閾値処理、モルフォロジー処理を組み合わせるほうが調整しやすい
- フレームをまたぐ塗りつぶしは、各フレームの閉領域ラベルを作り、隣接フレーム間で領域対応を取る処理が必要になるため、FFmpeg 単体より OpenCV ベースのほうが実装しやすい

## 5. 線画変換アルゴリズム

### 候補

#### A. 単純なエッジ抽出のみ

- 実装は軽い
- ただしノイズが多く、BBS っぽい整理された線になりにくい

#### B. エッジ抽出 + 二値化 + 細線化 + 解像度制御

- 画作りを調整しやすい
- 「ドット感はあるがドット絵ではない」という要件に寄せやすい

#### C. 学習ベース線画化

- 見た目は強い可能性がある
- ただしローカルアプリでの同梱サイズ、推論速度、再現性の負担が大きい

### 採用

`エッジ抽出 + 二値化 + 細線化 + 低解像度化/再拡大`

### 選定理由

- 要件はアニメ調の高品質線画生成ではなく、意図的に「古いお絵かき BBS 感」を残すこと
- そのため、超解像や ML よりも、解像度を少し落としてから線を整理し、表示時に補間を切るほうが要件に合う
- 調整パラメータが明確で、実装初期から試行錯誤しやすい

## 6. フレーム編集 UI

### 候補

#### A. HTML Canvas 2D

- 利点
  - ピクセル編集と相性が良い
  - `putImageData` や `imageSmoothingEnabled = false` を直接使える
  - 編集系 UI を素直に組みやすい
- 欠点
  - 大量オブジェクト管理は自前になりやすい

#### B. PixiJS / WebGL

- 利点
  - 表示性能は出しやすい
- 欠点
  - 今回は「ゲーム的描画」より「編集ツール」であり、管理対象が画像レイヤ中心なので少し過剰

#### C. SVG ベース

- 利点
  - ベクタ系 UI には向く
- 欠点
  - フレーム単位のラスター編集には不向き

### 採用

`React + TypeScript + HTML Canvas 2D`

### 選定理由

- このアプリの編集対象はベクタではなくラスター画像
- キャンバス 2D API のほうがブラシ、バケツ、オニオンスキン、ズーム表示を素直に実装できる
- BBS 風の荒い表示は補間無効化で再現しやすい
- React はツールバー、タイムライン、パネル状態管理に使い、描画自体は Canvas に寄せる分離がよい

## 7. フレーム横断の塗りつぶし方式

### 候補

#### A. 各フレーム独立の flood fill

- 実装は簡単
- 仕様を満たさない

#### B. オプティカルフロー追跡

- 動きへの追従性はある
- 閉領域の保証が弱く、仕様に対して過剰

#### C. 領域ラベリング + フレーム間対応付け

- 各フレームで閉領域を connected components として抽出
- 隣接フレームで重なり率、重心距離、面積変化を使って同一領域系列をつなぐ
- 仕様の「形が変わっても閉じていれば塗れる」に最も素直

### 採用

`領域ラベリング + フレーム間対応付け`

### 選定理由

- 要件はモーション推定ではなく、閉じた領域の継続判定
- そのため、領域単位の追跡のほうが仕様と一致する
- 将来的に「この塗りはどの区間まで伝播したか」を UI で見せやすい

## 8. 内部データ形式

### 候補

#### A. フレーム画像群 + JSON メタデータ

- 利点
  - デバッグしやすい
  - 壊れても復旧しやすい
- 欠点
  - ファイル数は増える

#### B. SQLite 一括管理

- 利点
  - 単一ファイルで扱いやすい
- 欠点
  - 画像実体を大量に持つと運用が重くなりやすい
  - 手で確認しにくい

### 採用

`プロジェクトディレクトリ方式`

- `source.mp4`
- `frames/line/000001.png` のような変換済みフレーム列
- `frames/paint/000001.png` のようなユーザー描画レイヤ
- `project.json` に fps、解像度、塗り情報、領域追跡情報を保存

### 選定理由

- 画像編集系では中間成果物を直接見られることが重要
- 不具合解析、再変換、キャッシュ破棄がしやすい
- 初期実装段階では DB 化より運用が単純

## 9. 最終採用スタック

- デスクトップ基盤: `Tauri`
- フロントエンド: `React + TypeScript`
- 描画: `HTML Canvas 2D`
- バックエンド処理: `Rust`
- 動画入出力: `FFmpeg`
- 画像処理: `OpenCV`
- プロジェクト保存: `PNG 連番 + JSON`

## 10. 補足方針

- 初版では GPU 前提にしない
- 線画変換は「完全自動高品質」より「調整しやすいパラメータ設計」を優先する
- 先にオフライン変換を終えてから編集することで、UI の応答性を守る

## 11. 編集画面アーキテクチャ方針

Tauri を採用しても、編集画面が高速になる保証はありません。重要なのは、`WebView 上では軽い表示と入力処理だけを担当させる` ことです。

### 11.1 基本方針

- React は `UI 状態管理` に限定する
- 実際の描画は `Canvas 2D` に寄せる
- 動画処理と画像解析は `Rust + FFmpeg + OpenCV` に寄せる
- フレーム実体の受け渡しは `巨大 JSON` ではなく `ファイルパス` または `バイナリ` を使う
- 編集対象は常に `現在フレーム + 前後数フレーム` だけをメモリ展開する

### 11.2 責務分離

#### フロントエンド側で持つもの

- 現在フレーム番号
- 再生状態
- 選択中ツール
- ブラシサイズ、選択色
- ズーム率、スクロール位置
- タイムラインの表示状態

#### Rust 側で持つもの

- 動画の読込と分解
- 線画変換
- 閉領域ラベリング
- フレーム横断の領域対応
- `mp4` の再エンコード
- 画像キャッシュの読み書き

#### Canvas 側で持つもの

- 現在表示中フレームの合成結果
- オニオンスキン用の前後フレーム
- 一時ブラシストローク
- プレビュー表示

### 11.3 画面構成

初版は以下の 3 ペイン構成が妥当です。

1. 左ペイン
   - ツール選択
   - 色選択
   - 線画変換パラメータ
2. 中央ペイン
   - メインキャンバス
   - オニオンスキン表示
   - ズーム/パン
3. 下ペイン
   - フレームタイムライン
   - 現在フレーム周辺のみサムネイル表示

### 11.4 遅くしないための実装ルール

#### React に乗せないもの

- 各ピクセルデータ
- 毎フレーム更新される描画バッファ
- ブラシ移動中の全座標列
- 全フレーム分のサムネイル実体

これらを React state に入れると、再レンダリング負荷で即座に重くなります。

#### Canvas 描画ルール

- 線画レイヤと着色レイヤを分ける
- 再描画は `requestAnimationFrame` 単位で束ねる
- 描きかけストロークは別レイヤに描き、本体コミット時だけマージする
- 補間は無効化して BBS 風表示を維持する
- ズーム時も元画像を毎回再生成せず、既存ビットマップを拡大表示する

#### データ読み込みルール

- 全フレームを起動時にメモリへ載せない
- 現在フレームの前後だけを先読みする
- サムネイルは別解像度で事前生成する
- 元フレームと表示用フレームを分ける

### 11.5 推奨データフロー

#### 動画読込時

1. `mp4` を選択
2. Rust 側で `ffmpeg` によりフレーム分解
3. Rust 側で OpenCV 前処理を実行
4. 線画 PNG 群と領域メタデータを保存
5. フロントエンドへは `プロジェクト情報` と `必要なファイルパス` だけ返す

#### 編集時

1. 現在フレーム変更
2. 対象フレーム画像を読み込む
3. Canvas で線画レイヤと着色レイヤを合成表示
4. ブラシまたは塗りつぶし操作を適用
5. 変更結果を該当フレーム画像へ保存
6. 必要なら近傍フレームのキャッシュを更新

#### 書き出し時

1. 線画レイヤと着色レイヤをフレームごとに合成
2. Rust 側で画像列を `ffmpeg` に渡す
3. `mp4` を出力

### 11.6 塗りつぶし機能の UI/処理分離

この機能は最も重くなりやすいため、フロントエンドでは完結させないほうがよいです。

- フロントエンドは `どのフレームのどの座標を塗ったか` だけ送る
- Rust 側で該当領域 ID を特定する
- 隣接フレームに対して継続領域を探索する
- 対象フレーム群の着色レイヤを書き換える
- フロントエンドは結果の更新通知だけ受ける

これにより、重いループ処理や大量メモリアクセスを UI スレッドから切り離せます。

### 11.7 初版で避けるべき実装

- 全フレームの画像を base64 文字列で IPC する
- ブラシを動かすたびに Rust と同期通信する
- React state に `ImageData` を保持する
- タイムライン全体を常時フル解像度サムネイルで描く
- 1 枚の巨大キャンバスに全フレームを並べる

### 11.8 初版の性能目標

初版では以下を目標にすると現実的です。

- 1080p / 数百フレーム規模で編集可能
- 現在フレーム切替は体感待ちが短いこと
- ブラシ描画は追従遅延を感じにくいこと
- タイムラインスクロールが引っかからないこと

高 fps のリアルタイム動画編集を目指すより、`フレーム編集ツールとして快適` を目標に置くべきです。

### 11.9 結論

`Tauri は十分採用可能だが、Web 技術だから速い/遅いではなく、WebView 上に重い責務を載せない設計が必須` です。

この仕様では、以下を守れば大きな懸念は下げられます。

- 表示は `Canvas 2D`
- UI 状態は `React`
- 重処理は `Rust`
- 動画入出力は `FFmpeg`
- 領域解析は `OpenCV`
- フレームは `必要分だけ読み込む`

## 12. MVP の画面構成

初版では、機能を増やすよりも `動画を読み込み、線画化し、フレームを塗れて、mp4 に戻せる` ところまでを確実に通すべきです。

### 12.1 画面一覧

#### A. プロジェクト開始画面

- `mp4` の選択
- 新規プロジェクト作成
- 変換開始

この画面では編集機能を持たせず、入力ファイル選択と変換開始だけに絞ります。

#### B. 変換進捗画面

- フレーム分解の進捗
- 線画変換の進捗
- 領域解析の進捗
- 中断または完了通知

重い初期処理があるため、無反応に見せない専用画面を置いたほうがよいです。

#### C. 編集画面

- 中央にメインキャンバス
- 左にツールパネル
- 下にタイムライン
- 上にプロジェクト操作と書き出し操作

MVP の中心画面です。

#### D. 書き出しダイアログ

- 出力先指定
- fps 確認
- 解像度確認
- `mp4` 書き出し開始

### 12.2 編集画面で MVP に含めるもの

- フレーム移動
- 1 フレーム単位のブラシ描画
- 色選択
- フレーム横断塗りつぶし
- オニオンスキンのオン/オフ
- ズーム、パン
- 保存
- `mp4` 書き出し

### 12.3 編集画面で MVP から外すもの

- レイヤの複数管理

- 音声編集
- リアルタイム再生しながらの高頻度編集
- 複雑なアンドゥ履歴の永続化

初版では、`線画の上に塗る` という主目的から外れる機能を意図的に落としたほうがよいです。

### 12.4 編集画面レイアウト

#### ヘッダー

- プロジェクト名
- 保存
- 書き出し
- 現在フレーム情報

#### 左サイドバー

- ツール切替
  - ペン
  - 塗りつぶし
  - スポイト
- 色選択
- ブラシサイズ
- オニオンスキン設定

#### 中央キャンバス領域

- 線画レイヤ表示
- 着色レイヤ表示
- ブラシプレビュー
- ズームとパン

#### 下部タイムライン

- 現在フレーム周辺のサムネイル
- フレーム移動
- 塗り適用済みフレームの簡易表示

## 13. MVP のモジュール分割

初版で責務を切る単位は、`描画 UI`、`アプリ状態`、`プロジェクト IO`、`動画処理`、`領域処理` の 5 つで十分です。

### 13.1 フロントエンドモジュール

#### `ui/app-shell`

- 全体レイアウト
- ルーティング
- ダイアログ表示

#### `ui/editor`

- 編集画面の構成管理
- パネル配置
- ツール切替 UI

#### `ui/canvas`

- メインキャンバス描画
- オニオンスキン合成
- ブラシプレビュー
- ポインタ入力処理

#### `ui/timeline`

- フレームサムネイル表示
- 現在フレーム選択
- 近傍フレームの仮想化表示

#### `state/editor-store`

- 現在フレーム番号
- 選択ツール
- 色
- ズーム率
- 再描画トリガ

React の state はここで持つが、ピクセル配列は持たない前提です。

#### `infra/tauri-api`

- Tauri command 呼び出し
- エラー変換
- パス受け渡し

### 13.2 Rust バックエンドモジュール

#### `project`

- プロジェクト作成
- `project.json` 読み書き
- ディレクトリ構成管理

#### `video_io`

- `ffmpeg` 呼び出し
- フレーム分解
- 動画書き出し

#### `lineart`

- OpenCV 前処理
- 線画抽出
- 二値化
- 細線化

#### `region_tracking`

- 閉領域ラベリング
- フレーム間対応付け
- 塗り区間の探索

#### `paint`

- 指定フレームへの描画反映
- 塗りつぶし結果の書き込み
- 着色レイヤ保存

#### `cache`

- 表示用フレームの読み出し
- サムネイル生成
- 近傍フレームの先読み

### 13.3 フロントと Rust の境界

フロントエンドから Rust へは、以下のような粒度で command を切るのがよいです。

- `create_project(video_path)`
- `get_project(project_path)`
- `get_frame_bundle(frame_index)`
- `draw_stroke(frame_index, stroke_data)`
- `fill_region(frame_index, x, y, color)`
- `export_mp4(output_path)`

この粒度なら、UI 側は操作意図だけを渡し、重い処理は Rust 側へ閉じ込められます。

### 13.4 フレーム表示データの単位

`get_frame_bundle(frame_index)` では、最低限以下を返す設計がよいです。

- 線画フレームのパス
- 着色レイヤのパス
- 前後フレームの参照情報
- サイズ情報
- サムネイル参照情報

ここで画像本体を巨大な JSON として返す必要はありません。

## 14. 実装順序

MVP を崩さないためには、以下の順で作るのが安全です。

1. プロジェクト作成と動画分解
2. 線画変換パイプライン
3. 編集画面のキャンバス表示
4. 1 フレーム描画

## 18. 受け入れ条件

各マイルストーンの完了判定を曖昧にしないため、最低限の受け入れ条件を定義します。

### 18.1 M1 受け入れ条件

- 1 本の `mp4` を読み込み、プロジェクトディレクトリが生成される
- `project.json` に fps、幅、高さ、総フレーム数が保存される
- `frames/source/` に連番 PNG が出る

### 18.2 M2 受け入れ条件

- 任意のフレームを開いて描画できる
- 保存後にアプリを開き直しても描画結果が残る
- フレーム切替時に UI が極端に固まらない

### 18.3 M3 受け入れ条件

- 閉じた領域をクリックすると、継続区間の複数フレームへ塗りが反映される
- 形が多少変化しても、閉領域が継続していれば塗りがつながる
- 明らかに別領域へ誤伝播しにくい

### 18.4 M4 受け入れ条件

- 編集結果を含む `mp4` が出力できる
- 出力動画を一般的な動画プレイヤーで再生できる
- 元の fps と長さが大きく崩れない

## 20. Tauri command 案

Rust 側 API は、初版では以下の command があれば十分です。

- `create_project(video_path: String, project_root: String) -> ProjectSummary`
- `open_project(project_root: String) -> ProjectSummary`
- `start_preprocess(project_root: String) -> TaskId`
- `get_preprocess_progress(task_id: String) -> ProgressState`
- `get_frame_bundle(project_root: String, frame_index: u32) -> FrameBundle`
- `draw_stroke(project_root: String, frame_index: u32, stroke: StrokeInput) -> SaveResult`
- `fill_region(project_root: String, frame_index: u32, x: u32, y: u32, color: Rgba) -> FillResult`
- `export_video(project_root: String, output_path: String) -> TaskId`

type FrameBundle = {

## 21. 主な技術リスク

### 21.1 線画変換の見た目が狙い通りにならない

- リスク
  - 単なるエッジ画像になり、BBS らしい整理感が出ない
- 対応
  - パラメータを `preset` 化する
  - 低解像度化と再拡大を前提に調整する

### 21.2 フレーム横断塗りつぶしが誤伝播する

- リスク
  - 近い別領域を同一と判定してしまう
- 対応
  - 面積差、重心距離、重なり率の複合スコアにする
  - しきい値を設定ファイル化する

### 21.3 画像 IO がボトルネックになる

- リスク
  - フレーム切替時にディスク待ちが増える
- 対応
  - 近傍フレーム先読み
  - サムネイル事前生成
  - 表示解像度と保存解像度を分ける

### 21.4 書き出し時のメモリ使用量が増える

- リスク
  - 全フレーム一括ロードでメモリを使い切る
- 対応
  - ストリーム処理で順次合成して `ffmpeg` に渡す

## 22. 実装開始時の優先ルール

初手で守るべきルールは以下です。

- 画像本体を React state に入れない
- 動画処理をフロントエンドに寄せない
- まず 1 本の動画で最後まで通す
- 最適化は `フレーム切替` と `ブラシ追従` に集中する
- 仕様の中核は `フレーム横断塗りつぶし` であることを忘れない
